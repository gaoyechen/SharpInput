## 意图确认示例

### 示例 1 — 输入："RAG和Fine-tuning怎么选"

```
问题: "你在比较 RAG 和 Fine-tuning，你的核心关注点是？"
  header: "意图确认"
  选项:
    - "我想了解各自的适用场景和边界" : 输出结构化对比，不给推荐
    - "我有具体数据和场景，需要推荐" : 输出决策建议，附带约束条件
    - "我想看一个成本/效果的量化分析" : 输出带数据支撑的分析框架
    - "其他" : 描述你的具体需求
```

### 示例 2 — 输入："帮我写个简历"

```
问题: "你说的「写简历」，核心诉求是？"
  header: "意图确认"
  选项:
    - "我需要一份可以直接用的简历模板" : 生成型，输出成品

    - "我有简历草稿，想请你帮我优化" : 分析型，诊断+改进
    - "其他" : 描述你的具体需求
```

## 语义模式详细信号

### 意图 1: 理解
- ❶ 输入是一个名词/概念，且没附带约束条件（"什么是DDD"）
- ❷ 用"为什么"开头但不涉及具体异常（"为什么用Redis" ≠ "为什么Redis报错"）
- ❸ 期望解释而非行动

### 意图 2: 决策
- ❶ 输入包含两个或多个互斥选项
- ❷ 描述一个需要取舍的十字路口
- ❸ 没有现成方案，要做选择

### 意图 3: 生成
- ❶ 期望产出一个不存在的产物（代码/文档/方案/设计）
- ❷ 从零开始，不基于已有对象

### 意图 4: 分析
- ❶ 提供了一个已有对象（方案/数据/现状）
- ❷ 期望得到结构化判断，不是解释也不是选择

### 意图 5: 探索
- ❶ 处于开放空间，没有明确收敛方向
- ❷ 期望拓宽视野而非收窄选择

### 意图 6: 诊断/调试
- ❶ 描述一个异常行为或预期不符的结果
- ❷ 隐含"哪里出了问题"的归因需求

### 意图 7: 说服/沟通
- ❶ 目标受众不是AI而是另一个人/群体
- ❷ 核心挑战是影响他人而非获取信息

### 意图 8: 验证/确认
- ❶ 已经包含一个结论/方案/判断
- ❷ 寻求检验而非生成

### 意图 9: 规划
- ❶ 有明确起点和终点
- ❷ 期望路径设计而非方向选择

### 意图 10: 学习/习得
- ❶ 用"教"而非"告诉我"，暗示需要可练习的知识而非信息罗列
- ❷ 涉及技能习得，不只是概念理解

### 意图 11: 优化/改进
- ❶ 基于已有对象的改进，不是从零开始
- ❷ 现状"能用但不满意"，期望变得更好
- ❸ 边界规则：当用户使用"怎么优化/改进/提升" → 主意图=优化；当用户使用"为什么/怎么回事/哪里出了问题" → 主意图=诊断；当两者混合 → 优化为主，诊断为次

### 意图 12: 比较/对比
- ❶ 输入包含两个或多个可比较对象
- ❷ 未明确是"要选一个"（→决策）还是"想搞清楚"（→理解）

### 意图 13: 总结/梳理
- ❶ 输入是一堆信息/文档/内容
- ❷ 期望压缩和结构化，不是分析也不是生成

### 意图 14: 求助/救援
- ❶ 表达了困境但没有明确的下一步期望
- ❷ 真实意图可能是诊断、决策或生成——必须穿透

## 打破映射规则的详细示例

| 信号 | 动作 | 示例 |
|------|------|------|
| 输入指向"正确但没用"的答案路径 | 至少 1 个维度从批判挑战类强制选取 | "怎么提升团队效率"→默认是 system/minimalist，应加入 `adversarial` |
| 输入包含强烈但未被质疑的前提 | 必须包含 `hidden-assumption` | "AI 取代程序员"→前提是线性替代，维度必须包含 hidden-assumption |
| 输入情绪信号强烈（焦虑/愤怒/无奈） | 优先 `role-reversal` 或 `time-horizon`，不做冷分析 | "公司又要裁员了"→不用 `structure` 做冷分析 |
