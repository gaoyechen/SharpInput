# Judge Agent Component Tests

> 测试目标：验证 Level 3 Judge Agent 的独立审查能力——包括反方辩护、真实反例、翻转条件和风险判定。

## Test Structure

每条测试包含：
- **Original Question**: 用户原始问题
- **Path A/B**: 两条优化路径（含完整优化问题）
- **Expected Checks**: Judge 应正确执行的检查项
- **Expected Risk**: 期望每条路径的风险判定

---

## TC-J-01 — 标准二路径审查

**Original Question**: `"K8s和Docker Swarm怎么选"`
**Path A** (risk-first): "请以「魔鬼代言人」的视角回答——K8s和Docker Swarm选型时，不要列优缺点，先说每个方案在什么条件下会崩溃。假设团队只有3个人，哪个方案的安全边界更宽？请给出反直觉的风险分析和真实的失败案例时间线。"
**Path B** (time-horizon): "请以「实战操盘手」的视角回答——站在1年后的时间点回看今天的选型决策。如果选择K8s，前3个月的运维成本是多少？如果Docker Swarm，半年后扩展时会遇到什么瓶颈？去他们的官网检查现有方案，并给出基于时间的路径图。"

**Expected Checks**:
- ✅ 反方辩护：每个路径3个具体论据，能被反驳
- ✅ 反方防御：用当前路径攻击另一条路径
- ✅ 真实反例：至少 1 条路径有实例（如"某公司选型失败"）
- ✅ 翻转条件：可量化（如"团队 ≥ 5人"）
- ⛔ 没有"可能/也许/某些情况下"模糊语

## TC-J-02 — 反例缺失→标记"未验证"

**Original Question**: `"用Golang重构Python后端划算吗"`
**Path A**: (counter-intuitive) "请以「魔鬼代言人」的视角回答——假设Golang比Python快5倍，但开发慢2倍，什么时候总成本会超过收益？找到Python的瓶颈到底是语言还是架构。"
**Path B**: (structure) "请以「领域专家」的视角回答——从系统架构角度分析，你见过的最成功的Golang重构案例和最失败的案例分别是什么？关键区别在哪？"

**Expected Checks**:
- ✅ 如果 Judge 找不到真实案例 → 写"未验证"（不编造）
- ⛔ 不用"某公司""某产品"模糊表述
- ✅ 报告格式严格包含 `=== JUDGE REPORT ===`

## TC-J-03 — 三条路径

**Original Question**: `"AI编程会取代程序员吗"`
**Path A** (hidden-assumption): "请以「魔鬼代言人」的视角回答——「AI会取代程序员」的最大前提是什么？如果这个前提不成立，结论会怎么反转？给出当前最被忽视的3个反例。"
**Path B** (time-horizon): "请以「实战操盘手」的视角回答——站在2030年回看，哪些编程工作没有被AI替代？哪些被替代的实际上是「编码」不是「编程」？给出可观测的信号。"
**Path C** (role-reversal): "请以「跨界猎人」的视角回答——从产业史视角看，每一次「X会取代Y」的预言在什么条件下被自我推翻？参考汽车取代马车的S曲线，AI编程现在处于哪个阶段？"

**Expected Checks**:
- ✅ 3条路径都需 Judge 审查
- ✅ Path C 输出（当 PATH_COUNT=3）
- ✅ 每个路径独立风险判定

## TC-J-04 — 风险判定一致性

**Original Question**: `"副业应该做SaaS还是自媒体"`
**Path A**: "请以「魔鬼代言人」的视角回答——如果SaaS副业在6个月内没有达到100个付费用户，你的沉没成本是多少？自媒体在6个月内没有1000粉丝时，策略是否完全失效？"
**Path B**: "请以「实战操盘手」的视角回答——不要问哪个方向更好，问「如果只有每周10小时，哪个方向三个月后能给你第一个可量化的结果」。给出本周的具体行动。"

**Expected Checks**:
- ✅ 风险判定使用 可靠/有条件/高风险 三者之一
- ✅ 判断标准一致（同一条路径不会因 Judge 换表述而改变判定）
- ✅ 有条件时说明了翻转条件可控程度

## TC-J-05 — Judge Agent 降级（模拟失败）

**Original Question**: `"如何提高团队代码质量"`
**Path A**: "请以「魔鬼代言人」的视角回答——代码质量的根本矛盾不是「写得好不好」而是「要不要上线」。找出一条路径：质量提升 50% 但交付速度下降 80%，什么时候这个取舍是错的？"
**Path B**: "请以「实战操盘手」的视角回答——不要给Code Review/自动化测试/规范文档三件套。反过来想：哪一条最常用措施在你团队实际上没起作用？为什么？"

**Simulated Failure #1**（Judge 首次失败）：
- 应操作：通知用户 → 简化版重试
- 简化版检查：去掉了反方防御和精确格式，保留反方攻击（2论据）+ 真实反例 + 翻转条件 + 风险判定

**Simulated Failure #2**（Judge 再次失败）：
- 应操作：降级为内联审查
- 内联检查：主代理自行标注"关键假设"和"风险标注"
- 输出标注：`独立审查服务未响应，已降级为内联审查。风险判定标的 (*)，仅供参考。`

## TC-J-06 — 硬约束执行

**Original Question**: `"要不要裸辞搞创业"`
**Path A**: "请以「魔鬼代言人」的视角回答——假设你的创业项目在6个月内零收入，你的储蓄能撑多久？如果答案 < 6个月，这个决策的根本风险是什么？"
**Path B**: "请以「实战操盘手」的视角回答——不是问「要不要」而是问「如果一定要，最安全的入场姿态是什么」。全职和兼职创业的真实概率差距在哪里？"

**Expected Hard Constraints**:
- ⛔ Judge 输出不能包含：可能/或许/某些情况下/一般来说/理论上
- ✅ 反方论据具体到可被反驳
- ✅ Judge 回答不是路径的作者，不辩护只攻击

# 执行指南

1. 对每条 TC 执行 Judge 流程（Agent 工具调用）
2. 记录 Judge 是否成功返回
3. 检查报告格式（=== JUDGE REPORT ===）
4. 验证每个字段的完整性和可靠性
5. 测试降级路径是否按规则执行
6. Judge 通过率目标: ≥ 80% (6/7)
